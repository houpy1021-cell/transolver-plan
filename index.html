基于 Transolver 的二维流道压力场预测方案

1. 问题定义

本阶段先考虑二维流道中的压力场预测问题。目标是建立一个数据驱动的算子模型，使其能够根据入口边界流动信息和流道几何边界信息，预测流道内部任意查询点处的压力。

数学形式可以写成：

p_i = F_theta(B_inlet, G_wall, x_i, y_i)

其中：

B_inlet = 入口边界点云
G_wall  = 流道几何边界点云
(x_i, y_i) = 内部查询点坐标
p_i = 查询点处的压力

也就是说，模型学习的是：

入口边界条件 + 几何边界 -> 内部压力场

而不是简单的点到点插值。

2. 数据形式

2.1 入口点云

入口处取 m 个点，每个点包含坐标和入口流动状态。

第一版建议使用：

inlet point = (x, y, p, T, u, v)

其中：

x, y = 入口点坐标
p    = 入口静压或给定压力
T    = 入口温度
u, v = 二维速度分量

如果目前只有速度大小，也可以先写成：

inlet point = (x, y, p, T, V)

但从物理上看，速度分量 (u, v) 比速度大小 V 更好，因为它保留了流动方向信息。

入口点云整体形状：

B_inlet in R^{m x 6}

2.2 边界点云

流道边界取 n 个点，每个点描述几何边界的位置和方向。

建议使用：

wall point = (x, y, nx, ny)

其中：

x, y   = 边界点坐标
nx, ny = 边界单位法向量

这里建议使用 nx, ny 这个命名，而不是 partial_x, partial_y，因为后者容易被理解成偏导数。边界法向量可以帮助模型理解壁面的朝向，也方便后续加入无穿透等物理约束。

边界点云整体形状：

G_wall in R^{n x 4}

2.3 内部查询点

内部取 q 个查询点。查询点不是输出真值，而是告诉模型需要在哪些位置预测压力。

基础形式：

query point = (x, y)

查询点整体形状：

X_query in R^{q x 2}

2.4 输出

第一阶段只预测压力：

P_query in R^{q x 1}

即：

每个内部查询点输出一个压力值 p

3. 为什么查询点也要作为输入

模型需要预测的是连续空间场：

p = p(x, y)

如果只输入入口点云和边界点云，模型不知道应该在哪些位置输出压力，也不知道输出点的数量和空间分布。因此查询点坐标 (x, y) 必须作为输入。

这里输入查询点并不是把输出压力提前告诉模型。模型输入的是：

query token = (x, y)

模型输出的是：

p

所以没有信息泄漏。查询点只是问题的位置，压力才是答案。

4. 总体实验路线

建议按照逐步增强的方式开展实验：

V1: 基础 Transolver
V2: Transolver + cross attention
V3: Transolver + cross attention + 物理信息嵌入

这样可以清楚比较每个模块的贡献，而不是一开始把所有设计全部堆进去。

5. V1: 基础 Transolver

5.1 设计目的

第一版目标是验证 Transolver 能否作为主干模型，学习：

入口边界条件 + 几何边界 + 查询点坐标 -> 查询点压力

这一版不显式加入 cross attention，也不加入复杂物理约束，重点是先跑通基础框架。

5.2 Token 设计

由于入口点、边界点和查询点的原始特征维度不同，需要先通过不同的 embedding 层投影到统一隐藏维度 d_model。

inlet_tokens = MLP_inlet(x, y, p, T, u, v)
wall_tokens  = MLP_wall(x, y, nx, ny)
query_tokens = MLP_query(x, y)

对应形状：

inlet_tokens in R^{m x d_model}
wall_tokens  in R^{n x d_model}
query_tokens in R^{q x d_model}

为了让模型区分不同 token 的来源，加入 type embedding：

inlet_tokens = inlet_tokens + type_inlet
wall_tokens  = wall_tokens  + type_wall
query_tokens = query_tokens + type_query

然后在 token 维度拼接：

all_tokens = concat(inlet_tokens, wall_tokens, query_tokens)

形状为：

all_tokens in R^{(m+n+q) x d_model}

5.3 模型流程

B_inlet  -> MLP_inlet  -> inlet tokens
G_wall   -> MLP_wall   -> wall tokens
X_query  -> MLP_query  -> query tokens

inlet tokens + wall tokens + query tokens
        -> concat
        -> Transolver backbone
        -> take query tokens only
        -> MLP head
        -> predicted pressure

5.4 损失函数

基础版只对查询点压力计算监督损失：

L_data = MSE(p_pred, p_true)

只取 query tokens 的输出参与损失计算，入口 token 和边界 token 不参与压力预测损失。

6. V2: Transolver + Cross Attention

6.1 加入 Cross Attention 的原因

基础 Transolver 中，入口、边界和查询点是直接拼接后共同混合的。这样虽然可以交互，但“谁向谁取信息”的结构不够明确。

本问题本质上是：

内部查询点从入口边界和几何边界中提取信息

因此可以引入 cross attention，使模型结构更加符合物理问题：

query = 内部待预测点
context = 入口点云 + 边界点云

6.2 Token 设计

仍然先分别 embedding：

inlet_tokens = MLP_inlet(x, y, p, T, u, v) + type_inlet
wall_tokens  = MLP_wall(x, y, nx, ny)       + type_wall
query_tokens = MLP_query(x, y)              + type_query

将入口 token 和边界 token 拼成 context：

context_tokens = concat(inlet_tokens, wall_tokens)

形状：

context_tokens in R^{(m+n) x d_model}
query_tokens   in R^{q x d_model}

6.3 Cross Attention 关系

cross attention 中：

Q = query_tokens
K = context_tokens
V = context_tokens

含义是：

每个内部查询点，根据自己的空间位置，从入口点和边界点中选择性提取有用信息。

输出仍然对应查询点：

query_tokens_updated in R^{q x d_model}

6.4 模型流程

B_inlet -> MLP_inlet -> inlet tokens
G_wall  -> MLP_wall  -> wall tokens
X_query -> MLP_query -> query tokens

inlet tokens + wall tokens
        -> concat
        -> context tokens

query tokens attend to context tokens
        -> cross attention
        -> updated query tokens
        -> Transolver blocks
        -> MLP head
        -> predicted pressure

也可以先对 context 做若干 Transolver block，再让 query cross-attend：

context tokens -> Transolver encoder -> encoded context
query tokens cross-attend encoded context
query tokens -> Transolver decoder -> pressure

6.5 预期优势

相比 V1，V2 的结构更明确：

入口点和边界点负责提供条件信息
查询点负责提出预测位置
cross attention 负责边界到内部的信息映射
Transolver 负责空间 token 之间的场信息传播

7. V3: 加入物理信息嵌入

7.1 设计原则

这一阶段不采用完整 PINN，不直接加入 Navier-Stokes 残差。先采用更稳定的物理信息嵌入方式：

1. 物理特征输入
2. 无量纲化变量
3. 几何相关辅助量
4. 轻量物理约束

这样可以让模型从纯数据驱动变成 physics-embedded data-driven model，同时避免 PINN 训练困难的问题。

7.2 入口 token 的物理增强

基础入口特征：

(x, y, p, T, u, v)

物理增强后可以使用：

(x_norm, y_norm, p_norm, T_norm, u_norm, v_norm, Mach)

其中：

Mach = sqrt(u^2 + v^2) / a
a = sqrt(gamma * R * T)

如果暂时没有完整热物性，也可以先用速度无量纲量替代 Mach。

7.3 边界 token 的物理增强

基础边界特征：

(x, y, nx, ny)

增强后：

(x_norm, y_norm, nx, ny, wall_type)

其中：

wall_type = 边界类型标识

例如：

0 = lower wall / hub
1 = upper wall / casing

如果后续有入口、出口边界，也可以继续扩展。

7.4 查询 token 的物理增强

基础查询特征：

(x, y)

增强后建议使用：

(x_norm, y_norm, d_wall_norm, eta, A_ratio)

其中：

d_wall = 查询点到最近壁面的距离
eta = d_lower / (d_lower + d_upper)
A_ratio = H(x) / H_in

含义：

d_wall 反映查询点离壁面的距离
eta 反映查询点在上下壁面之间的相对高度
A_ratio 反映局部流道高度或面积相对入口的变化

这些特征不属于 PINN，但它们具有明确物理意义，可以帮助模型理解变几何流道中的空间位置。

7.5 输出变量的物理化

不建议直接预测原始压力数值，可以先预测无量纲压力，例如：

p_norm = (p - p_ref) / p_scale

或者压力系数：

Cp = (p - p_ref) / (0.5 * rho_ref * V_ref^2)

第一版可以采用更简单稳定的形式：

p_norm = (p - mean(p_train)) / std(p_train)

后续再替换成基于入口参考状态的物理无量纲形式。

7.6 轻量物理损失

如果后续希望进一步加强物理约束，但又不想做 PINN，可以加入轻量物理 loss：

L = L_data + lambda_wall * L_wall + lambda_inlet * L_inlet

其中：

L_data = 查询点压力监督损失
L_inlet = 入口附近预测压力与入口给定压力的一致性

如果模型后续预测速度场，可以加入壁面无穿透：

L_wall = ||V dot n||^2

但当前只预测压力时，暂时不加入速度相关的壁面损失。

8. 推荐实施顺序

阶段 1: 跑通基础 Transolver

输入：

inlet: (x, y, p, T, u, v)
wall:  (x, y, nx, ny)
query: (x, y)

输出：

p

模型：

三类 token 分别 embedding -> concat -> Transolver -> query head -> p

目的：

验证 Transolver 主干是否能学习基本压力场映射。

阶段 2: 加入 Cross Attention

输入不变。

模型：

入口 token + 边界 token -> context
查询点 token -> query
query cross-attend context
Transolver 继续处理 query tokens
MLP head 输出压力

目的：

显式建模边界条件和几何信息到内部查询点的传递。

阶段 3: 加入物理信息嵌入

输入增强为：

inlet: (x_norm, y_norm, p_norm, T_norm, u_norm, v_norm, Mach)
wall:  (x_norm, y_norm, nx, ny, wall_type)
query: (x_norm, y_norm, d_wall_norm, eta, A_ratio)

输出：

p_norm 或 Cp

目的：

通过物理特征和无量纲化提高模型泛化能力，同时避免完整 PINN 的训练复杂度。

9. 方案总结

本方案将二维流道压力预测问题建模为点云算子学习问题。入口边界点云提供流动条件，壁面边界点云提供几何信息，内部查询点提供预测位置。基础 Transolver 用于建立点云 token 之间的全局信息传播；cross attention 用于显式表达内部查询点从入口和边界 context 中提取信息；物理信息嵌入则通过无量纲化、法向量、距壁距离、相对通道位置和面积比等特征，将工程物理先验融入数据驱动模型。

推荐从最简单的基础 Transolver 开始，逐步加入 cross attention 和物理信息嵌入，通过消融实验比较不同模块的贡献。
